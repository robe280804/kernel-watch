🛡️ Security monitoring made visible

KernelWatch

The silent guardian that watches your Docker containers from inside the Linux kernel — and warns you the moment one of them starts behaving like it has been broken into.

KernelWatch watches what every program actually does on your server — not what it claims to do. When a container suddenly opens a shell, reads a password file, scans the network, or installs new tools, KernelWatch notices in real time and raises an alert with a clear severity and a plain-language reason.

👀 Where it watches from

The trick that makes it powerful — and impossible to hide from.

🧩

Nothing inside your containers

You don't install anything in your apps. A single guard runs on the host. Your containers stay untouched.

🔬

It listens at the kernel

Every action a program takes (open a file, run a command, connect somewhere) passes through the Linux kernel. KernelWatch listens exactly there.

🥷

An attacker can't turn it off

Even if someone takes over a container, they can't see or disable a monitor that lives below them, in the kernel.

🚦 The journey of a single event

From "a program did something" to "you get an alert" — in milliseconds.

Something happens

A program in a container runs a command, opens a file, or opens a network connection.

▼

The kernel reports it

KernelWatch is notified instantly, with no delay and no missed events.

▼

Which container was it?

It figures out which container the action came from. Actions on the host itself are ignored unless you switch on host monitoring.

▼

Who launched it? (the smart part)

It looks at the family tree of the program — who started it. This is what tells a routine task apart from an intruder. (See below.)

▼

You get an alert

If it looks dangerous, an alert is raised with a severity, a reason, and the MITRE ATT&CK category — then sent to your log, database, Slack or webhook.

🧠 The key idea: context decides everything

The exact same command can be perfectly normal or a serious incident. KernelWatch judges by who launched it, not just what it is. This is what keeps it quiet on normal days and loud during a real attack.

✓ Normal — stays quiet
sh -c "php artisan schedule:run"

A shell started by your scheduled task. Expected. KernelWatch recognises the trusted parent and stays silent.

Family tree: scheduler → php → sh ✓ trusted
⚠ Attack — critical alert
sh -c "id; cat /etc/passwd"

The same kind of shell — but started by your public web server. That almost always means someone broke in through the website.

Family tree: nginx → php-fpm → sh ⚠ internet-facing

Same command. Opposite verdict. That's the difference between a tool that floods you with noise and one you actually trust.

🌡️ How serious is it?

Every alert carries one of four levels, so you know what needs you now.

Low — worth noting, rarely urgent
Medium — suspicious, review when you can
High — likely malicious, look soon
Critical — almost certainly an attack, act now

You choose the minimum level you want to hear about. By default you get Medium and above; quieter "Low" events are dropped.

🔗 Connecting the dots: attack chains

Real attacks aren't one event — they're a sequence. A shell opens, then the network is scanned, then secrets are stolen, then something is planted to stay.

KernelWatch watches for these sequences inside the same container. When several suspicious events line up like a real intrusion, it raises one consolidated "attack chain" alert with a risk score — instead of burying you in a dozen separate ones. These are never silenced.
